Overview

REST API for managing users and books.

Base URL

http://localhost:5000/api

Endpoints

  • POST /auth/signup
  • POST /auth/login
  • GET /auth/me
  • GET /books
  • GET /books/:id
  • POST /books
  • PUT /books/:id
  • DELETE /books/:id
  • GET /search?q=
  • GET /admin/users
  • GET /admin/events
  • GET /admin/stats

Authentication

POST /auth/signup

Create a new user.

FieldType
namestring
emailstring
passwordstring

Response: 201 Created

POST /api/auth/signup { "name": "John", "email": "john@test.com", "password": "123456" }
POST /auth/login

Login user and receive token.

FieldType
emailstring
passwordstring

Response: 200 OK (JWT Token)

GET /auth/me

Get current user.

Headers: Authorization: Bearer token

Response: 200 OK

Books

GET /books

Get all books.

Response: 200 OK

GET /api/books
GET /books/:id

Get a single book.

Response: 200 OK or 404

POST /books

Add a new book.

FieldType
titlestring
authorstring
coverstring (URL)
datedate
pageCountnumber
genrestring

Response: 201 Created

{ "_id": "123", "title": "Atomic Habits", "author": "James Clear", "cover": "https://...", "date": "2018-01-01", "pageCount": 320, "genre": "Self-help" }
PUT /books/:id

Update a book.

Response: 200 OK

DELETE /books/:id

Delete a book.

Response: 200 OK

Admin

GET /admin/users

Get all users.

Auth Required: Admin

Response: 200 OK

GET /admin/events

Get admin activity logs.

Response: 200 OK

GET /admin/stats

Get system stats (users, books, activity).

Response: 200 OK

GET /search?q=

Search books, users, and activity.

Query: q (string)

Response: 200 OK

Database (MongoDB)

This API uses MongoDB to store users and books.

Book Schema

{ _id: ObjectId, title: String, author: String, cover: String, date: Date, pageCount: Number, genre: String }

User Schema

{ _id: ObjectId, name: String, email: String, password: String (hashed), role: String (user | admin), createdAt: Date }

Example Query

db.books.find({ genre: "Fantasy" })

Notes

  • Passwords are hashed before storing
  • MongoDB automatically generates _id
  • Relationships are handled using IDs
